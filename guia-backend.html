Cómo usar el backend de inicio a fin

Esta guía está pensada para trabajar con Swagger sin perderse entre tantos endpoints. Sigue el flujo recomendado y usa las secciones de decisión cuando llegues a un punto donde se puede hacer más de una cosa.

1. Idea general

La plataforma tiene 5 microservicios detrás de Kong:

usuarios

Login, registro, personas, usuarios y roles. Firma los JWT.

zonas

Zonas y espacios del parqueadero. Controla si un espacio está disponible u ocupado.

vehiculos

Vehículos por cliente. Permite buscar por placa para operar en puerta.

asignaciones

Relación oficial entre vehículo y propietario.

tickets

Entrada, salida, pago, cancelación y valor recaudado.

Todo se consume por Kong: http://localhost:8000. No llames directo a los backends internos.

2. Arranque diario

Primera vez o si faltan secretos

bash scripts/bootstrap.sh

Esto genera/prepara:

  • .env
  • .secrets/jwt-private.pem
  • .secrets/jwt-public.pem
  • infrastructure/kong/kong.yml

Levantar todo

docker compose up -d --build

Confirmar estado

docker compose ps

Busca que estén healthy: usuarios, zonas, vehiculos, asignaciones, tickets, kong y web.

Si cambiaste .env, claves JWT o Kong, vuelve a correr bash scripts/bootstrap.sh y recrea los servicios con docker compose up -d --build.

3. Swagger y token

Swagger centralizado:

http://localhost:8000/asignaciones/swagger-ui

Desde el selector superior puedes cambiar entre Usuarios, Zonas, Vehículos, Asignaciones y Tickets.

Cómo leer Swagger sin perderte

Swagger muestra nombres técnicos como persona-controller o user-controller. No son pasos de negocio, solo son carpetas donde Swagger agrupa endpoints.
Si ves esto en SwaggerPiensa estoCuándo abrirlo
auth-controllerLogin, registro y sesión actualCuando necesitas entrar, registrar cliente o probar tu token.
user-controllerUsuarios del sistemaCuando ADMIN crea usuarios o asigna roles.
role-controllerRolesCuando necesitas copiar el ID de RECAUDADOR, ADMIN, etc.
persona-controllerDatos personalesCuando ADMIN busca una persona por cédula, nombre o correo.
vehiculosCarros, motos, placasCuando CLIENTE crea vehículos o RECAUDADOR busca por placa.
ticketsEntrada, salida, pago y cancelaciónCuando RECAUDADOR trabaja en la puerta.

Cómo iniciar sesión

  1. En Swagger selecciona Usuarios.
  2. Busca el bloque auth-controller. Ese bloque significa autenticación.
  3. Abre POST /api/v1/auth/login.
  4. Presiona Try it out.
  5. Pega el JSON de login.
  6. Presiona Execute.
{
  "username": "admin",
  "password": "Admin12345!"
}

Copia el accessToken. Luego presiona Authorize y pega:

TU_ACCESS_TOKEN
En Swagger pega solo el token. No pegues Bearer, comillas, el JSON completo ni el refreshToken.
Para comprobar que estás autenticado, en el mismo bloque auth-controller ejecuta GET /api/v1/auth/me.

4. Roles

RolPara qué sirveQué puede hacer
CLIENTE Persona que usa el parqueadero. Ver zonas/espacios, crear sus vehículos, ver sus tickets y asignaciones.
RECAUDADOR Empleado de puerta. Buscar vehículos, crear tickets, pagar tickets y cancelar tickets.
ADMIN Administrador funcional. Gestionar usuarios, roles, zonas, espacios, vehículos, asignaciones y tickets.
ROOT Superusuario técnico. Puede usar rutas /root/** para borrado físico.
El microservicio se llama usuarios, pero eso no significa que exista el rol viejo USER. El rol funcional normal ahora es CLIENTE.

5. Iniciar sesión por rol

Cada vez que cambies de rol, vuelve a ejecutar login, copia el nuevo accessToken y reemplázalo en el botón Authorize de Swagger.

ADMIN Login válido para administrar

Cuándo usarlo: crear zonas, espacios, usuarios, recaudadores, consultar personas por cédula, revisar todo.
Paso: Usuarios, bloque de login, endpoint POST /api/v1/auth/login 
{
  "username": "admin",
  "password": "Admin12345!"
}
Después del login

Copia solo accessToken, pulsa Authorize y pégalo sin Bearer.

Prueba rápida

Ejecuta GET /api/v1/usuarios. Si lista usuarios, estás como ADMIN.

RECAUDADOR Login válido para operar puerta

Cuándo usarlo: buscar vehículo por placa, crear ticket, pagar ticket, cancelar ticket, ver espacios disponibles.
Paso: Usuarios, bloque de login, endpoint POST /api/v1/auth/login 
{
  "username": "rdrecaudador",
  "password": "Recaudador12345!"
}
Después del login

Copia solo accessToken, pulsa Authorize y reemplaza el token anterior.

Prueba rápida

Ejecuta GET /api/v1/vehiculos/placa/PAC-0004 o GET /api/v1/tickets.

CLIENTE Login válido para usuario normal

Cuándo usarlo: ver sus propios vehículos, asignaciones y tickets; crear vehículo propio; ver zonas y espacios.
Paso: Usuarios, bloque de login, endpoint POST /api/v1/auth/login 
{
  "username": "adalvarez",
  "password": "Demo12345!"
}
Después del login

Copia solo accessToken, pulsa Authorize y reemplaza el token anterior.

Prueba rápida

Ejecuta GET /api/v1/vehiculos. Debe mostrar solo vehículos de ese cliente.

ROOT Login para borrado físico

Importante: en la base demo actual no dejé una cuenta ROOT lista para usar. ROOT se usa solo para rutas /api/v1/root/** y borrado físico. Para pruebas normales usa ADMIN.
Si necesitas ROOT

Primero debes crear un usuario, buscar el rol ROOT con GET /api/v1/roles y asignarlo con POST /api/v1/usuarios/{userId}/roles/{roleId}. Hazlo solo si el profesor pide probar borrado físico.

6. Flujo completo recomendado

Usa esta guía por casos de uso, no como una lista larga. Cada caso tiene su propia pestaña con lo previo, el servicio de Swagger, el endpoint, el JSON y qué dato guardar.

Para trabajar: abre 7. Casos de uso en Swagger. Para demostrar validaciones y errores: abre 8. Pruebas de validación.

7. Casos de uso en Swagger

Entra por http://localhost:8000/asignaciones/swagger-ui. Cambia el servicio arriba y sigue estos flujos.

ADMIN Caso 1: entrar y dejar Swagger autorizado

Objetivo: entrar como ADMIN para poder administrar usuarios, roles, zonas, espacios y revisar datos.
Paso 1: ve al servicio Usuarios

En Swagger, arriba, selecciona Usuarios. Luego abre el bloque auth-controller, que es el bloque de login.

Paso 2: abre POST /api/v1/auth/login

Presiona Try it out y coloca:

{
  "username": "admin",
  "password": "Admin12345!"
}
Paso 3: copia accessToken y autoriza Swagger

Copia solo accessToken y pegalo. No pegues Bearer, comillas ni refreshToken.

Paso 4: prueba que el token sirve

Ejecuta para comprobar que el token quedo bien. Debe responder el usuario admin.

GET /api/v1/auth/me

CLIENTE Caso 2: registrar un cliente

Objetivo: crear un cliente normal. Al registrarse queda con rol CLIENTE automáticamente.
Paso 1: ve a Usuarios y abre registro

En Swagger selecciona Usuarios, abre auth-controller y entra a POST /api/v1/auth/register.

Paso 2: presiona Try it out y pega el JSON

Esto crea persona, usuario y rol CLIENTE automáticamente.

{
  "persona": {
    "dni": "1712345678",
    "firstName": "Juan",
    "middleName": null,
    "lastName": "Perez",
    "email": "juan.perez@test.com",
    "phone": "0999999999",
    "address": "Cuenca",
    "nationality": "Ecuatoriana"
  },
  "password": "Cliente12345!"
}
Paso 3: guarda estos datos de la respuesta

user.idPerson es el ID del cliente. user.username es el nombre con el que hara login. El accessToken ya viene con rol CLIENTE.

Cliente demo disponible: adalvarez / Demo12345!.

ADMIN Caso 3: crear un recaudador

Objetivo: crear un empleado de puerta. Este caso necesita ADMIN porque un usuario normal no puede darse el rol RECAUDADOR a sí mismo.
Paso previo: entra como ADMIN

Primero haz el Caso 1 y deja Swagger autorizado con token ADMIN.

Paso 1: crear el usuario empleado

En Swagger selecciona Usuarios, abre el bloque user-controller y entra a POST /api/v1/usuarios. Esto crea el usuario, pero todavía no le da rol RECAUDADOR.

{
  "persona": {
    "dni": "1723456789",
    "firstName": "Carlos",
    "middleName": null,
    "lastName": "Recaudador",
    "email": "carlos.recaudador@test.com",
    "phone": "0991234567",
    "address": "Cuenca",
    "nationality": "Ecuatoriana"
  },
  "password": "Recaudador12345!"
}

Guarda idPerson como userId y guarda el username generado.

Paso 2: buscar el ID del rol RECAUDADOR

En el mismo servicio Usuarios, abre role-controller y ejecuta:

GET /api/v1/roles

Busca "name": "RECAUDADOR" y copia su id como roleId.

Paso 3: asignar el rol RECAUDADOR al usuario

Vuelve a user-controller y abre:

POST /api/v1/usuarios/{userId}/roles/{roleId}

En userId pega el idPerson. En roleId pega el ID del rol RECAUDADOR. Si Swagger pide body, pon:

{}
Paso 4: probar login del recaudador

Abre auth-controller y ejecuta POST /api/v1/auth/login.

Prueba el login con el username generado y la contraseña que pusiste.

Recaudador demo disponible: rdrecaudador / Recaudador12345!.

ADMIN Caso 4: crear zona y espacio

Objetivo: crear un lugar físico donde luego el RECAUDADOR pueda emitir tickets.
Paso previo: entra como ADMIN

Haz el Caso 1 y deja Swagger autorizado con token ADMIN.

Paso 1: crear una zona

En Swagger selecciona Zonas. Busca el bloque de zonas y abre POST /api/v1/zonas.

{
  "nombre": "Zona Prueba",
  "descripcion": "Zona creada desde Swagger",
  "tipo": "REGULAR",
  "capacidad": 10
}

Guarda el id devuelto. Ese es idZona.

Paso 2: crear un espacio dentro de esa zona

En el mismo servicio Zonas, busca el bloque de espacios y abre POST /api/v1/espacios.

{
  "codigo": "P-001",
  "idZona": "ID_DE_LA_ZONA",
  "descripcion": "Espacio de prueba para auto",
  "tipo": "AUTO",
  "capacidad": 1,
  "estado": "DISPONIBLE"
}

Guarda el id devuelto. Ese es idEspacio para tickets.

Resultado esperado: el espacio debe aparecer como DISPONIBLE cuando ejecutes GET /api/v1/espacios.

CLIENTE Caso 5: crear vehiculo propio

Paso previo: entra como CLIENTE

Entra como cliente. Puedes usar adalvarez / Demo12345!. Pega su token en Authorize.

Paso 1: crear el vehículo

En Swagger selecciona Vehículos y abre POST /api/v1/vehiculos.

{
  "tipo": "auto",
  "datos": {
    "placa": "ABC-1234",
    "marca": "Toyota",
    "modelo": "Corolla",
    "color": "Gris",
    "anio": 2022,
    "clasificacion": "Gasolina",
    "numeroPuertas": 4,
    "capacidadMaletero": 470
  }
}

Guarda el id devuelto como vehicleId.

No mandes ownerId. El dueño sale del token del CLIENTE.

CLIENTE Caso 6: asignar vehiculo a propietario

Objetivo: dejar registrado oficialmente que ese vehículo pertenece a un cliente. Sin esto, Tickets no puede saber quién es el dueño.
Paso previo: tener un vehículo creado

Primero haz el Caso 5 y guarda el id del vehículo.

Paso 1: asignar el vehículo usando token CLIENTE

En Swagger selecciona Asignaciones y abre POST /api/v1/asignaciones.

{
  "vehicleId": "ID_DEL_VEHICULO"
}
Opción alternativa: si estás como ADMIN 
{
  "userId": "ID_DEL_CLIENTE",
  "vehicleId": "ID_DEL_VEHICULO"
}
Sin asignacion activa, Tickets no puede saber de quien es el vehiculo.

RECAUDADOR Caso 7: registrar entrada con ticket

Objetivo: registrar que un vehículo entra al parqueadero, ocupar el espacio y crear ticket ACTIVO.
Paso previo: entra como RECAUDADOR

Entra como rdrecaudador / Recaudador12345! y pega su token en Authorize.

Paso 1: buscar el vehículo por placa

En Swagger selecciona Vehículos y abre GET /api/v1/vehiculos/placa/{placa}.

Prueba con una placa disponible del seed:

PAC-0004

Resultado esperado: te devuelve el vehículo. Si no existe, debe salir 404 en español.

Paso 2: buscar un espacio disponible

En Swagger selecciona Zonas y abre:

GET /api/v1/espacios

Busca un espacio con "estado": "DISPONIBLE" y copia su id.

Paso 3: emitir el ticket

En Swagger selecciona Tickets y abre POST /api/v1/tickets.

{
  "idEspacio": "ID_DEL_ESPACIO_DISPONIBLE",
  "placa": "PAC-0004"
}

Guarda el id del ticket. Debe quedar ACTIVO.

Paso 4: verificar ocupación del espacio

Ejecuta GET /api/v1/espacios. El espacio debe quedar OCUPADO.

RECAUDADOR Caso 8: registrar salida y cobrar

Objetivo: cerrar la estadía, calcular el valor con mínimo de 30 minutos y liberar el espacio.
Paso previo: entra como RECAUDADOR

Debe existir un ticket ACTIVO. Puedes usar uno creado en el Caso 7.

Paso 1: buscar ticket activo

En Swagger selecciona Tickets y ejecuta:

GET /api/v1/tickets?estado=ACTIVO

Busca el ticket activo y copia su id.

Paso 2: pagar el ticket

Abre PATCH /api/v1/tickets/{id}/pagar.

Pega el ID en el path y ejecuta. No necesitas body.

Paso 3: revisa la respuesta

Debe quedar PAGADO, con fechaHoraSalida y valorRecaudado. El espacio vuelve a DISPONIBLE.

RECAUDADOR Caso 9: demostración rápida: crear, listar y cobrar

Objetivo: este es el flujo que puedes mostrar cuando te pidan literalmente “crear, cobrar y listar tickets”. Usa un vehículo sin ticket activo y un espacio DISPONIBLE.
Paso previo: entrar como RECAUDADOR

En Swagger selecciona Usuarios, abre login y entra con:

{
  "username": "rdrecaudador",
  "password": "Recaudador12345!"
}

Copia solo el accessToken, presiona Authorize y pégalo.

Paso 1: listar tickets antes de crear

En Swagger selecciona Tickets y ejecuta:

GET /api/v1/tickets

Esto demuestra que el sistema puede listar tickets. También puedes usar:

GET /api/v1/tickets?estado=ACTIVO
Paso 2: buscar datos para crear el ticket

En Vehículos, busca la placa:

GET /api/v1/vehiculos/placa/PAC-0004

Luego en Zonas, lista espacios y copia un id con estado DISPONIBLE:

GET /api/v1/espacios
Paso 3: crear ticket

En Tickets, abre POST /api/v1/tickets y coloca:

{
  "idEspacio": "ID_DEL_ESPACIO_DISPONIBLE",
  "placa": "PAC-0004"
}

Resultado esperado: estado ACTIVO. Copia el id del ticket creado.

Paso 4: listar para comprobar que existe

Ejecuta de nuevo:

GET /api/v1/tickets?estado=ACTIVO

Debe aparecer el ticket que acabas de crear.

Paso 5: cobrar ticket

En Tickets, abre PATCH /api/v1/tickets/{id}/pagar. En id pega el ID del ticket creado.

No pongas body. Solo presiona Execute.

Paso 6: listar para comprobar el cobro

Ejecuta:

GET /api/v1/tickets

Resultado esperado: el ticket ahora debe salir con estado PAGADO, fechaHoraSalida y valorRecaudado.

RECAUDADOR Caso 10: cancelar ticket

Objetivo: anular una entrada creada por error, sin cobrar y liberando el espacio.
Paso previo: entra como RECAUDADOR y ten un ticket ACTIVO

Si no tienes uno, crea un ticket con el Caso 7.

Paso 1: cancelar el ticket

En Swagger selecciona Tickets y abre PATCH /api/v1/tickets/{id}/cancelar.

Usa esto si el ticket fue un error o no corresponde cobrar. Solo funciona si esta ACTIVO.

Resultado esperado

Debe quedar CANCELADO, valorRecaudado: 0.00 y espacio DISPONIBLE.

CLIENTE Caso 11: revisar lo propio

Paso previo: login como CLIENTE

Usa adalvarez / Demo12345! y pega su token en Authorize.

Paso 1: endpoints que sí puede usar 
GET /api/v1/vehiculos
GET /api/v1/asignaciones
GET /api/v1/tickets
GET /api/v1/zonas
GET /api/v1/espacios
Paso 2: endpoints que deben fallar con 403 
POST /api/v1/zonas
DELETE /api/v1/vehiculos/{id}
PATCH /api/v1/tickets/{id}/pagar

8. Pruebas de validación para tener a mano

Usa estos casos cuando quieras demostrar que el backend valida placa, propietario, ticket activo, espacio disponible y estados. Recomendado: haz login como rdrecaudador / Recaudador12345! para tickets, y como admin / Admin12345! cuando necesites ver personas o cédulas.

Validación 1: buscar vehículo por placa

Previo requerido: iniciar sesión como RECAUDADOR, ADMIN o ROOT y pegar el accessToken en Authorize. Debe existir al menos un vehículo en la base. Con datos demo usa PAC-0004.
Vehículos -> GET /api/v1/vehiculos/placa/{placa}

Con token de RECAUDADOR, prueba una placa demo:

PAC-0004

Resultado esperado: devuelve el vehículo. Guarda su id si quieres emitir ticket por idVehiculo.

Placa inexistente

Prueba:

ZZZ-9999

Resultado esperado: 404 con mensaje de vehículo no encontrado.

Validación 2: si ingreso por cédula, listar vehículos del cliente

Previo requerido: iniciar sesión como ADMIN. Debe existir un cliente con vehículos y asignaciones activas. Con datos demo puedes usar el cliente adalvarez.
Hoy el RECAUDADOR no tiene endpoint directo para buscar personas por cédula. En Swagger se valida con ADMIN: primero encuentras la persona por DNI y luego consultas su flota.
Paso 1: buscar la persona por cédula

En Swagger selecciona Usuarios, abre el bloque de personas y ejecuta GET /api/v1/personas.

Con token ADMIN, busca en la lista la cédula en el campo dni. Copia idUuid.

Cliente demo útil: busca el registro de adalvarez o una cédula que hayas registrado.

Paso 2: listar vehículos de esa persona

En Swagger selecciona Asignaciones y abre GET /api/v1/propietarios/{userId}/vehiculos.

En userId pega el idUuid de la persona/usuario.

Resultado esperado: lista de vehículos de ese cliente.

Regla mental: por placa encuentras un vehículo puntual; por cédula encuentras el cliente y luego su lista de vehículos.

Validación 3: si ingreso por placa, el backend resuelve el propietario

Previo requerido: iniciar sesión como RECAUDADOR. Debe existir un vehículo con asignación activa y un espacio DISPONIBLE. Con datos demo usa una placa disponible como PAC-0004.
Paso: emitir ticket desde el servicio Tickets

En Swagger selecciona Tickets y abre POST /api/v1/tickets.

Con token RECAUDADOR, usa una placa disponible y un espacio DISPONIBLE:

{
  "idEspacio": "ID_DE_UN_ESPACIO_DISPONIBLE",
  "placa": "PAC-0004"
}
Resultado esperado

El ticket queda ACTIVO y la respuesta trae idUsuario. Ese idUsuario es el propietario resuelto desde la asignación activa del vehículo.

La respuesta de ticket muestra idUsuario, no la cédula. Si necesitas ver la cédula en Swagger, entra como ADMIN y consulta GET /api/v1/personas/{idUsuario}.

Validación 4: emitir ticket válido

Previo requerido: iniciar sesión como RECAUDADOR. Debes tener una placa con propietario activo y sin ticket activo, más un espacio DISPONIBLE. Guarda antes el idEspacio desde GET /api/v1/espacios.
Paso 1: buscar espacio disponible

En Swagger selecciona Zonas y ejecuta GET /api/v1/espacios.

Copia un espacio con estado: DISPONIBLE.

Paso 2: emitir ticket desde Tickets

En Swagger selecciona Tickets y abre POST /api/v1/tickets.

{
  "idEspacio": "ID_DEL_ESPACIO_DISPONIBLE",
  "placa": "PAC-0004"
}

Guarda el id del ticket, el idEspacio y la placa usada.

3. Confirmar ocupación

Ejecuta GET /api/v1/espacios. El espacio usado debe quedar OCUPADO.

Validación 5: intentar emitir otro ticket con el mismo vehículo

Previo requerido: primero ejecuta la Validación 4 y deja el ticket en estado ACTIVO. No pagues ni canceles todavía. Necesitas la misma placa usada en ese ticket.
Paso: intentar emitir otro ticket

En Swagger selecciona Tickets y abre POST /api/v1/tickets.

Usa la misma placa del ticket activo, aunque cambies el espacio:

{
  "idEspacio": "OTRO_ESPACIO_DISPONIBLE",
  "placa": "PAC-0004"
}
Resultado esperado

Debe responder 409: el vehículo ya tiene un ticket activo.

Validación 6: intentar usar el mismo espacio ocupado con otro vehículo

Previo requerido: primero ejecuta la Validación 4 para ocupar un espacio. Debes guardar ese idEspacio ocupado y tener otra placa disponible con asignación activa, por ejemplo PAD-0005.
Paso: intentar ocupar el mismo espacio

En Swagger selecciona Tickets y abre POST /api/v1/tickets.

Usa el mismo idEspacio que quedó OCUPADO, pero otra placa disponible, por ejemplo:

{
  "idEspacio": "MISMO_ESPACIO_OCUPADO",
  "placa": "PAD-0005"
}
Resultado esperado

Debe responder 409: el espacio no está disponible. Esta prueba valida la ocupación del espacio.

Validación 7: pagar ticket y liberar espacio

Previo requerido: debe existir un ticket ACTIVO. Puedes usar el ticket creado en la Validación 4. Guarda su id y el idEspacio para verificar que se libere.
Paso: pagar el ticket activo

En Swagger selecciona Tickets y abre PATCH /api/v1/tickets/{id}/pagar.

Usa el id del ticket ACTIVO que emitiste.

Resultado esperado

El ticket queda PAGADO, aparece valorRecaudado y el espacio vuelve a DISPONIBLE.

Prueba extra

Intenta pagar el mismo ticket otra vez. Debe responder 409 porque ya no está activo.

Validación 8: cancelar ticket y liberar espacio

Previo requerido: crea un ticket nuevo y déjalo ACTIVO. Usa una placa que no tenga otro ticket activo y un espacio DISPONIBLE. No uses un ticket ya pagado o cancelado.
1. Crea otro ticket ACTIVO

Usa otra placa disponible y otro espacio DISPONIBLE.

Paso 2: cancelar el ticket activo

En Swagger selecciona Tickets y abre PATCH /api/v1/tickets/{id}/cancelar.

Usa el id del ticket recién creado.

Resultado esperado

El ticket queda CANCELADO, valorRecaudado queda en 0.00 y el espacio vuelve a DISPONIBLE.

Validación 9: vehículo sin propietario activo

Previo requerido: crear un vehículo nuevo, pero no crearle asignación en /api/v1/asignaciones. Luego iniciar sesión como RECAUDADOR y usar un espacio DISPONIBLE.
1. Como CLIENTE o ADMIN, crea un vehículo nuevo

No crees asignación en /api/v1/asignaciones.

2. Como RECAUDADOR, intenta emitir ticket con esa placa 
{
  "idEspacio": "ID_DE_ESPACIO_DISPONIBLE",
  "placa": "PLACA_SIN_ASIGNACION"
}
Resultado esperado

Debe responder que el vehículo no tiene propietario activo asignado. Esta prueba valida asignaciones.

Datos demo útiles: PAB-0001 ya tiene ticket ACTIVO, PAC-0004 está disponible para probar, y puedes buscar espacios disponibles con GET /api/v1/espacios.

9. Usuarios: qué usar y cuándo

Para iniciar todo

Usa POST /api/v1/auth/login con admin.

Para crear un cliente

Usa POST /api/v1/auth/register. Este endpoint crea rol CLIENTE.

{
  "persona": {
    "dni": "1712345678",

Para crear un recaudador

  1. Consulta roles con GET /api/v1/roles.
  2. Busca el ID del rol RECAUDADOR.
  3. Crea usuario con POST /api/v1/usuarios o usa uno existente.
  4. Asigna rol con POST /api/v1/usuarios/{userId}/roles/{roleId}.
No uses root-controller para pruebas normales. Es para borrado físico y requiere rol ROOT.

10. Zonas y espacios

Crear zona

POST /api/v1/zonas
{
  "nombre": "Zona Norte",
  "descripcion": "Zona principal",
  "tipo": "REGULAR",
  "capacidad": 20
}

Crear espacio

POST /api/v1/espacios
{
  "codigo": "A-001",
  "idZona": "ID_DE_LA_ZONA",
  "descripcion": "Espacio para auto",
  "tipo": "AUTO",
  "capacidad": 1,
  "estado": "DISPONIBLE"
}

Para tickets, el espacio debe estar DISPONIBLE. Si está OCUPADO, RESERVADO o FUERA_DE_SERVICIO, tickets rechazará la entrada.

12. Asignaciones

Una asignación dice quién es el propietario activo de un vehículo. Tickets necesita esto para saber a qué cliente pertenece el ticket.

Si estás como CLIENTE

POST /api/v1/asignaciones
{
  "vehicleId": "ID_DEL_VEHICULO"
}

Si estás como ADMIN

{
  "userId": "ID_DEL_CLIENTE",
  "vehicleId": "ID_DEL_VEHICULO"
}
Si intentas crear un ticket sin asignación activa, tickets responderá que el vehículo no tiene propietario activo.

13. Tickets

Crear ticket de entrada

POST /api/v1/tickets

Por placa:

{
  "idEspacio": "ID_DEL_ESPACIO",
  "placa": "ABC-1234"
}

Por ID de vehículo:

{
  "idEspacio": "ID_DEL_ESPACIO",
  "idVehiculo": "ID_DEL_VEHICULO"
}

Al crear el ticket:

  • El ticket queda ACTIVO.
  • El espacio pasa a OCUPADO.
  • No se permite otro ticket activo para el mismo vehículo.

Pagar salida

PATCH /api/v1/tickets/{id}/pagar

Al pagar:

  • El ticket pasa a PAGADO.
  • Se calcula valorRecaudado.
  • El espacio vuelve a DISPONIBLE.

Cancelar

PATCH /api/v1/tickets/{id}/cancelar

Al cancelar:

  • El ticket pasa a CANCELADO.
  • valorRecaudado queda en 0.
  • El espacio vuelve a DISPONIBLE.

Fórmula de cobro

minutosCobrados = max(30, minutosEntreIngresoYSalida)
valor = (minutosCobrados / 60) * tarifaTipoVehiculo * factorTipoEspacio

14. Si tienes dos caminos

SituaciónCamino recomendadoAlternativa
Necesitas probar rápido. Login como ADMIN y crea datos base. Crear usuarios reales con roles separados.
Crear ticket. Usar placa, porque es lo natural en puerta. Usar idVehiculo si ya lo tienes desde Swagger.
Vehículo sale normalmente. PATCH /tickets/{id}/pagar. Cancelar solo si la operación fue error o no debe cobrarse.
Espacio aparece ocupado. Revisar si hay ticket ACTIVO. Como ADMIN, cambiar estado del espacio si quedó inconsistente.
Cambiaste código. docker compose up -d --build servicio. Reconstruir todo con docker compose up -d --build.
Datos quedaron mal. Corregir desde Swagger si es posible. docker compose down -v para borrar bases y empezar de cero.

15. Errores comunes

ErrorQué significaQué hacer
401 No enviaste token, expiró o está mal pegado. Si el mensaje dice Bad token, lo respondió Kong antes del backend. Haz login otra vez y pega solo el accessToken en Authorize. No pegues Bearer, comillas, JSON completo ni refreshToken.
403 Tu rol no tiene permiso. Revisa si estás como CLIENTE, RECAUDADOR o ADMIN según el endpoint.
404 No existe el recurso o el cliente no puede verlo. Verifica IDs y que el recurso pertenezca al usuario correcto.
409 Conflicto de negocio. Puede ser ticket activo duplicado, placa repetida o espacio no disponible.
Kong no muestra Tickets Kong no recargó su configuración. Ejecuta bash scripts/bootstrap.sh y luego docker compose restart kong.
Servicios no están healthy Algún backend sigue arrancando o falló. Usa docker compose logs -f --tail=100 servicio.

16. Comandos clave

Preparar secretos y Kong

bash scripts/bootstrap.sh

Levantar todo

docker compose up -d --build

Reconstruir solo tickets

docker compose up -d --build tickets

Ver estado

docker compose ps

Ver logs

docker compose logs -f --tail=100

Apagar sin borrar bases

docker compose down

Empezar desde cero borrando bases

docker compose down -v
docker compose up -d --build
Ruta diaria recomendada: abrir Swagger centralizado, hacer login en Usuarios, Authorize, crear datos base y operar Tickets.
